Сначала выделил самостоятельно проблемы из данных, затем попросил ИИ добавить свои идеи, которые так же глубоко обсудил и провалидировал, внес правки для финальных версий. Основная проблема в классификации вопроса: сделать либо через ключевые слова и регулярки, либо через классификацию через LLM + tool calling.

## Решения и альтернативы

| Решение | Альтернатива, которую отбросил | Почему |
| --- | --- | --- |
| UI: CLI | FastAPI, aiogram | Для работы с приложением из-за ужатых сроков, а fastapi либо aiogram = новые точки отказа, смысл задания показать работу с ИИ и построение системы, а не знание фреймворков |
| LLM провайдер: OpenAI | Anthropic, Mistral, Gemini | Личные проблемы с доступом, отстутствие кредитов, уже был знаком с API OpenAI |
| LLM: для саммари по запросу + советы из собранных данных | Самостоятельный сбор данных через ИИ | Большой контекст для ИИ, который можно вынести в обычные функции. Модель может придумать данные/не учесть что-то, ответы могут меняться от запроса к запросу. Через analytics.py мы полностью контролируем логику сбора данных, которая нужна для решения конкретных проблем, а так же даем нейросети сразу все самое необходимое для ответа |

## Промпты — как итерировались

### system.md

**v1:**
- коротко описал роль ассистента;
- запретил выдумывать факты;
- указал отвечать на русском и использовать только tool output.
- отдельно зафиксировал, что LLM не должна пересчитывать выручку, остатки и топы сама: все числа должны приходить из tools.
- добавил требование давать 1-3 практических действия только если они действительно следуют из данных.

Что заметил:
- модель может путать название клиента и контактное лицо, это нужно будет усилить в следующей версии промпта.
- модель в блоке "что сделать" предложила продолжить ответ следующим сообщением, хотя CLI не хранит историю диалога. Это создавало ложное ожидание, что можно написать "давай" или номер пункта и ассистент поймёт контекст.

**v2:**
- уточнил, что практические действия должны быть внешними бизнес-действиями: связаться с клиентом, проверить остаток, оформить дозаказ, исправить данные.
- запретил предложения вида "если хотите, я могу..." и продолжение "следующим сообщением", потому что текущая CLI-версия stateless и не хранит историю запросов.
- дополнительно запретил общие советы вроде "сверить учёт": если модель предлагает проверку, она должна назвать конкретные товары/клиентов/заказы из tool output и объяснить причину.
- добавил правило не просить пользователя пересчитывать готовую аналитику вручную, потому что расчёты уже делает слой аналитики.
- смягчил это правило для data quality случаев: если есть аномалия, модель может попросить проверить конкретную исходную строку или значение, указав id, поле и причину.
- уточнил поведение для количества действий: не добирать список до трёх, не выдумывать действия при пустых результатах и выбирать максимум 3 самых важных действия, если вариантов больше.
- добавил критерии приоритизации действий: риск потерять продажи, низкий остаток при спросе, крупный клиент, ошибка данных, влияющая на расчёты.

Что заметил:
- на мета-вопросы о возможностях ассистента модель всё равно добавляла блок практических действий, хотя бизнес-данные в таких запросах не используются.
- в приветствии/общем вопросе модель использовала расплывчатую формулировку "или что-то ещё", что может создать ожидание универсального ассистента.

**v3:**
- сформулировал общее правило: возможности ассистента ограничены доступными tools, а не заранее вручную перечисленными сценариями.
- запретил обещать действия, данные или сценарии, для которых нет инструмента или данных.
- указал, что на вопросы о возможностях и формате работы нужно отвечать кратко и без блока практических действий.
- запретил расплывчатые обещания вроде "или что-то ещё", если это не поддержано доступным инструментом.
- дополнительно уточнил, что условные предложения вроде "если нужен другой период..." не являются практическими действиями. Если из данных не следует внешнее бизнес-действие, блок действий лучше не добавлять.

**v4:**
- структурировал промпты по секциям role/context/instructions/output_format/examples и разнёс их на 4 файла: общий `system.md` + сценарии `client_profile.md`, `stock_risks.md`, `sales_summary.md`.
- использовал XML-теги как разметку границ секций, чтобы модель лучше отличала общие правила, контекст сценария, формат ответа и примеры.
- добавил few-shot примеры не для вычислений, а для граничных случаев: не путать клиента и контактное лицо, не придумывать товары при пустых рисках, не подменять "всю выручку" последними 30 днями.
- обновил загрузку промптов: первый LLM-вызов получает только `system.md` для выбора tool, а финальный ответ получает `system.md` + инструкцию только выбранного сценария.
- убрал дублирующие инструкции из сценарных промптов, потому что выбор сценария уже сделан через tool calling.
- сделал список практических действий открытым: примеры действий в `system.md` не должны ограничивать модель, если другое конкретное действие прямо следует из tool output.
- вынес negative constraints в отдельные `<constraints>` блоки внутри сценарных промптов, чтобы отделить "что делать" от "чего не делать".
- проверил идею фиксированного ответа при no-tool, но отказался от неё: для приветствий и вопросов о возможностях это делает UX слишком сухим.
- оставил no-tool ответы на уровне модели, но усилил `system.md`: вне домена ассистент должен вежливо отказаться, а на приветствие кратко назвать рамку возможностей.
- analytics слой вернул за весь доступный период `70` валидных completed-заказов и выручку `492043.95`, а модель в финальном ответе исказила эти значения до `69` и `483172.59`. Усилил `system.md` и `sales_summary.md`: числовые поля из tool output нужно переносить без изменения значения, нельзя округлять выручку, менять `orders_count`, период или пересобирать итог из отдельных строк
- привёл правило к общей схеме для всех сценариев: в `system.md` оставлен общий запрет на пересчёт фактов, а в сценарных промптах закреплены конкретные поля, которые нужно переносить из tool output без изменения.
- добавил аналогичные constraints для `client_profile.md` и `stock_risks.md`: модель не должна менять суммы, количество заказов, последний заказ, топы клиента, остатки, продажи за период, срок до дефицита и порядок рискованных товаров.
