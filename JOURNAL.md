## День 1 · [15.05]
### Сеанс 1 · [16:30-18:00]
- Что делал: смотрел исходные данные и думал над тем, какие можно выделить ключевые боли, чтобы эффективно решать их с помощью AI, сокращая время ручного труда сотрудников и нагрузку. Выделил 3 ключевые "боли", обсудил их с ИИ и финализировал. Выделил и обсудил архитектуру проекта с ИИ, пришел к разделению на слои: data_sources.py - получение raw данных, cleaning.py - модуль очистки и валидации данных, analytics.py - функции для подсчета, prompts.py - для загрузки промптов, tools.py - описание tools для модели, llm.py - модуль самой модели с главной функцией answer_question(). С помощью codex реализовал все слои, кроме tools.py, prompts.py и llm.py, провалидировал, вносил правки.
- Тупики: проблемы, которые выделяются, либо слишком сложные для MVP, либо слишком простые и не несут ценности, либо слишком завязаны на ИИ, что требует большего времени для внедрения и тестирования. Сначала выделил самостоятельно проблемы из данных, затем попросил ИИ добавить свои идеи, которые так же глубоко обсудил и провалидировал, внес правки для финальных версий. Следующий тупик, с которым столкнулся: как именно огранизовать архитектуру приложения. Необходимо было выбрать простой вариант из-за ужатых сроков, но при этом рабочий и с возможностью простого масштабирования. Основная проблема в классификации вопроса: сделать либо через ключевые слова и регулярки, либо через классификацию через LLM + tool calling.

### Сеанс 2 · [18:30-20:30]
- Что делал: с помощью codex + OpenAI API docs добавил llm function calling. Т.к. система обязательно должна работать через ИИ (иначе смысла нет), реализовал обычный function calling без роутинга с ключевыми словами + все равно нужно было бы доставать аргументы для функций из промпта, что было бы очень сложно через обычные регулярки, а в ответе мы бы получали обычные данные без советов ИИ. Первый system prompt самый простой для тестирования работоспособности приложения.

## Решения и альтернативы

| Решение | Альтернатива, которую отбросил | Почему |
| --- | --- | --- |
| UI: CLI | FastAPI, aiogram | Для работы с приложением из-за ужатых сроков, а fastapi либо aiogram = новые точки отказа, смысл задания показать работу с ИИ и построение системы, а не знание фреймворков |
| LLM провайдер: OpenAI | Anthropic, Mistral, Gemini | Личные проблемы с доступом, отстутствие кредитов, уже был знаком с API OpenAI |
| LLM: для саммари по запросу + советы из собранных данных | Самостоятельный сбор данных через ИИ | Большой контекст для ИИ, который можно вынести в обычные функции. Модель может придумать данные/не учесть что-то, ответы могут меняться от запроса к запросу. Через analytics.py мы полностью контролируем логику сбора данных, которая нужна для решения конкретных проблем, а так же даем нейросети сразу все самое необходимое для ответа |
