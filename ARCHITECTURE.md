## 1. Решения и почему

### Общий pipeline

```text
Подготовка данных при старте:
data_sources.py -> cleaning.py -> PreparedData

Обработка вопроса:
main.py -> llm.py -> tools.py -> analytics.py -> PreparedData
              |
              -> prompts.py
```

Это не совсем один линейный стек вызовов. `data_sources.py` и `cleaning.py` не являются входом пользовательского запроса, они заранее готовят `PreparedData`. После этого основной поток запроса идёт от `main.py` в `llm.py`, дальше модель выбирает tool, а tool вызывает нужную функцию из `analytics.py`, которая работает с уже подготовленными данными.

Если смотреть на логические зависимости слоёв, получается так:

```text
main.py -> llm.py -> tools.py -> analytics.py -> cleaning.py -> data_sources.py
```

`main.py` - входная точка CLI. Загружает `.env`, готовит данные один раз при старте и дальше принимает вопросы пользователя.

`data_sources.py` - слой получения сырых данных. Клиенты читаются из JSON, товары из Supabase, заказы из Google Docs. Если внешний источник недоступен, используется fallback для чтения данных из `test-data/`.

`cleaning.py` - слой подготовки данных. Он превращает сырые dict/CSV-строки в dataclass-модели `Client`, `Product`, `Order` и собирает `DataIssue`. Я не стал просто выбрасывать все проблемные строки, потому что в бизнес-данных даже плохая строка может быть полезным сигналом. Например, отрицательная сумма заказа не должна попасть в выручку, но должна быть показана как проблема данных.

`analytics.py` - детерминированный слой расчётов. Здесь считаются факты для трёх сценариев: профиль клиента, риски остатков, сводка продаж. LLM не считает выручку, топы и остатки самостоятельно, потому что это повышает риск галлюцинаций и нестабильных ответов.

`tools.py` - Здесь описаны JSON схемы для function calling и преобразование результатов dataclass в JSON. Благодаря этому модель может выбрать функцию и аргументы, но сами расчёты остаются в Python.

`llm.py` - слой общения с OpenAI API. Первый вызов модели выбирает tool и аргументы, затем Python выполняет tool, а второй вызов модели формирует понятный ответ пользователю на основе результата.

`prompts.py` и папка `prompts/` - загрузка промптов из файлов. Промпты вынесены из кода, потому что они являются частью поведения системы и должны версионироваться отдельно. Сейчас есть общий `system.md` и сценарные промпты для `client_profile`, `stock_risks`, `sales_summary`.

## 2. Альтернативы и почему отбросил

### CLI vs Telegram bot vs FastAPI

Я выбрал CLI, потому что для тестового важнее показать работу с данными, tool calling, промптами и качеством ответа. Telegram или FastAPI добавили бы больше точек отказа.

### Tool calling vs router по ключевым словам

Сначала рассматривал router по ключевым словам: если в вопросе есть “остатки”, вызвать функцию остатков, если “клиент” - профиль клиента. Отбросил, потому что нестабильно, сложно в реализации и в извлечении аргументов для функций, а если писать парсинг аргументов, то это получится такой же tool calling, только кривой. Сейчас модель сама выбирает функцию и извлекает аргументы, затем использует данные из analytics слоя.

### LLM считает всё сама vs Python считает факты

Можно было загрузить все данные в модель и попросить её считать ответы. Но модель может ошибиться в арифметике, пропустить грязные строки или каждый раз давать разный результат.

В текущем решении Python считает факты в `analytics.py`, а LLM только объясняет результат и формулирует практические действия, что делает систему более надежной и детерминированной.

### Один большой prompt vs system + scenario prompts

Сначала был один общий системный промпт. Потом стало понятно, что разные сценарии требуют разных ограничений: в клиентском профиле важно не путать `client.name` и `contact_person`, в остатках нельзя придумывать рискованные товары, в продажах нельзя менять `revenue` и `orders_count`.

Поэтому промпты разделены: `system.md` содержит общие правила, а сценарные файлы содержат конкретные инструкции и constraints для выбранного tool. Это уменьшает лишний контекст и упрощает поддержку.

### Жёсткая валидация vs мягкая очистка + DataIssue

Жёсткая валидация могла бы выбрасывать все строки с ошибками. Это проще для расчётов, но хуже для бизнес-сценария: пользователь не увидит, что в данных есть проблема.

Я выбрал мягкую очистку. Невалидные значения не участвуют в расчётах, если могут испортить результат, но сами проблемы сохраняются в `DataIssue` и могут быть показаны в ответе.

### Параллельные tool calls vs один выбранный tool

Параллельные вызовы могли бы лучше покрывать сложные комбинированные вопросы. Но для MVP я оставил один выбранный tool на запрос, чтобы проще отлаживать поведение и меньше рисковать смешиванием разных результатов.

## 3. Trade-offs решения

CLI легко запустить и проверить, но для реального использования Юлей и Ириной лучше подошёл бы Telegram-бот или простой веб-интерфейс.

Function calling гибче keyword-router, но поведение модели всё равно вероятностное. Она может выбрать не тот tool, неправильно извлечь период или исказить финальную формулировку. Из-за этого пришлось усиливать промпты и отдельно фиксировать, что числа из tool output нельзя пересчитывать.

Детерминированный `analytics.py` даёт контроль над фактами, но каждый новый бизнес-сценарий требует писать новую функцию, схему tool и сценарный промпт.

Мягкая очистка данных сохраняет больше информации, но усложняет верхние слои: нужно учитывать `None`, `DataIssue` и понимать, какие строки можно использовать в расчётах, а какие только показывать как проблему.

Разделение промптов по сценариям уменьшает лишний контекст для модели (так как в каждый вызванный tool мы загружаем только нужный промпт для этого сценария), но если добавить новый tool и забыть добавить сценарный prompt или constraints, качество ответа может просесть.

Система не хранит историю диалога. Это упрощает CLI и убирает проблемы состояния, но пользователь не может написать “а теперь подробнее по третьему пункту” и ожидать, что ассистент помнит прошлый ответ.

## 4. Где сломается на 100 пользователях

Каждый запуск CLI заново загружает данные и готовит их, из-за чего приложение может падать по OOM. Кэш может решить эту проблему.

Внешние источники могут быть медленными или недоступными, что заставит ждать пользователя ответа дольше обычного, либо сломает ответ без ретраев и обработок ошибок.

OpenAI API имеет rate limits, стоимость токенов и сетевые ошибки. При росте нагрузки нужны ретраи, лимиты, логирование ошибок и, возможно, кеширование одинаковых запросов.

При росте количества tools модель может чаще ошибаться с выбором функции. В OpenAI API есть параметр `tool_search`, который решает данную проблему при большом количестве функций, но она доступна только на новых моделях.

Нет обработки конкурентных запросов. CLI вообще не рассчитан на много пользователей одновременно.

Нет полноценного мониторинга качества. Сейчас можно вручную проверять ответы, но для реальной системы нужны golden questions, регулярный прогон `evaluate.py`, хранение результатов и анализ регрессий.

## 5. Что улучшил бы за ещё 3 дня

1. Довёл бы `tests/golden_questions.json` и `evaluate.py` до нормальной оценки качества.  
   Для AI-системы важно не только “ответило”, но и “ответило правильно”: выбрало нужный tool, не исказило числа, не придумало факты и дало полезное действие.

2. Добавил бы logging/trace для LLM-вызовов.  
   Нужно сохранять вопрос, выбранный tool, аргументы, tool output и финальный ответ. Это помогло бы быстрее понимать, где ошибка: в данных, analytics-слое, выборе tool или финальной формулировке модели (сделать это с CLI тяжелее, нужно разделять на `stderr` и `stdout`, легче реализовать с телеграм ботов или веб-интерфейсом).

3. Улучшил бы работу со сложными запросами и состоянием.  
   Сейчас система проще и stateless. Следующий шаг - поддержать историю диалога или аккуратные multi-tool сценарии, чтобы пользователь мог задавать уточнения и комбинировать несколько бизнес-вопросов.
